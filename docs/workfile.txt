2026-03-10  orchestrator/artifacts/jmx/server-normal.jmx
  - Increased operation duration_ms from 100 to 500 for all 3 operations (CPU, MEM, DISK)
  - Increased CPU intensity from 0.7 to 0.8
  - Root cause: 100ms operations were too brief to register on psutil.cpu_percent()
    (1-second sampling interval). 30 threads × 70ms CPU burn = operations complete before
    the next stats sample, resulting in 0.1% observed CPU. 500ms creates sustained
    overlapping load across all 4 cores.

2026-03-10  DB: load_profiles (id=1, "low")

  - Reason: on 4-vCPU VMs, thread granularity is coarse; narrow range causes
    calibration stability loop to exhaust

2026-03-11  orchestrator/artifacts/jmx/server-normal.jmx

    "Disk Mixed IO" → "disk"

2026-03-11  emulator/app/stats/collector.py

2026-03-12  orchestrator/src/orchestrator/core/calibration.py
  - BUG FIX: calibration stats returned 0% CPU because JMeter finished before
    stats were read (Run 27)
  - Root cause: JMeter duration_sec was exactly equal to orchestrator sleep time
    (observation: 30+15=45s sleep, 45s JMeter; stability: 60+15=75s sleep, 75s JMeter).

  - Fix: set JMeter duration_sec=3600 (1 hour) in both _run_observation() and
    _run_stability_check(). JMeter never self-terminates during calibration.

2026-03-12  orchestrator/src/orchestrator/infra/jmeter_controller.py
  - BUG FIX: JMeter start() hung for 120s (SSH timeout) after duration_sec=3600 change (Run 28)
  - Root cause: nohup command didn't redirect stdin (< /dev/null), so the backgrounded
    JMeter process inherited the SSH channel's stdin fd, keeping the channel open.
    recv_exit_status() blocked until the 120s SSH timeout, causing _run_observation()
    to fail with an empty exception on every iteration.
  - Fix: added < /dev/null to the nohup command so the SSH channel closes immediately
    after the shell prints the PID and exits.
  - Additional fix: added execute_background() method to SSHExecutor that reads stdout
    with a socket timeout instead of waiting for recv_exit_status() (which blocks when
    background processes keep the SSH channel open).

2026-03-12  orchestrator/src/orchestrator/core/baseline_orchestrator.py
  - BUG FIX: Pool sizing caused 99% CPU from GC pressure (Run 31)
  - Root cause: Pool was sized as % of total RAM (60% of 7.6GB = 4.5GB), but the JVM
    heap is total_ram - 2048MB = 5.6GB. A 4.5GB pool in a 5.6GB heap = 82% heap usage,
    leaving no room for GC. G1GC ran constantly at 389% CPU (all 4 vCPUs).
  - Fix 1: Changed pool sizing from % of RAM to % of JVM heap.
    _POOL_RAM_PERCENT -> _POOL_HEAP_PERCENT. server_steady: 50% of heap (was 60% of RAM),
    server_file_heavy: 30% of heap (was 40% of RAM). Heap is estimated as
    total_ram - 2048MB (matching start.sh formula).
  - Fix 2: Added _destroy_pool() and _cleanup_pools() methods. Pool is now released
    in the finally block of run(), ensuring cleanup on both success and failure.

2026-03-13  orchestrator/src/orchestrator/core/calibration.py
  - Replaced top-down binary search with ascending search (low-to-high).
  - Phase 1 "ramp_up": exponential ramp (1, 2, 4, 8, ...) until CPU exceeds
    target_max. JVM warms up naturally as load increases — no separate warmup
    phase needed.
  - Phase 2 "binary_search": refines between last_below and first_over if the
    exponential ramp overshot.
  - Removed _run_warmup() method and warmup_duration_sec config.
  - Reason: run 42 showed that high-thread warmup (15 threads) bloated JVM heap
    to 93% memory, causing 49% CPU at 1 thread (vs 24% in steady state). Going
    low-to-high avoids this — JVM memory grows proportionally to actual load.

2026-03-13  orchestrator/src/orchestrator/core/calibration.py
  - Replaced blind exponential doubling (1→2→4→8) with intelligent interpolation.
  - After each observation, uses linear model (CPU-per-thread) to predict the
    thread count needed for target midpoint, instead of blindly doubling.
  - With 1 data point: cpu_per_thread = observed/threads, predict target_mid/cpu_per_thread.
  - With 2+ data points: uses slope between last two observations.
  - When bracketed (last_below, first_over): uses linear interpolation between bounds.
  - Added _interpolate_candidate() and _predict_next_candidate() static methods.
  - Tracks full observation history: list of (thread_count, avg_cpu) tuples.
  - Reason: user requested data-driven approach. Exponential doubling is too coarse —
    e.g. 1→2 threads could overshoot from 25% to 60% on a 4-core VM. Intelligent
    prediction minimizes wasted calibration iterations.
